# 프로그래머스 코딩테스트 로드맵

## 목표

- 프로그래머스 기준으로 코딩테스트 합격 가능한 기초 실력 만들기
- 알고리즘 이론과 문제 풀이를 같이 가져가기
- Level 1에서 구현 실수를 줄이고, Level 2에서 자주 나오는 유형을 정복하기

## 전체 전략

프로그래머스 올인은 좋은 선택이다. 국내 코딩테스트와 문제 스타일이 가깝고, 특히 구현/문자열/해시/스택/큐/정렬/완전탐색/이분탐색/DFS-BFS 기초를 다지기에 적합하다.

다만 중요한 점은 "플랫폼"보다 "반복할 유형"이다.  
합격권 기초 실력은 어려운 한 문제를 오래 붙드는 것보다, 자주 출제되는 쉬운 유형을 빠르고 정확하게 푸는 능력에서 나온다.

## 6주 운영안

### 1주차
- 문자열
- 배열
- 구현
- 목표: Level 1 문제를 빠르게 읽고, 조건대로 정확히 구현하기

### 2주차
- 해시
- 스택
- 큐
- 목표: 자료구조 선택 감각 익히기

### 3주차
- 정렬
- 완전탐색
- 시뮬레이션
- 목표: 브루트포스와 정렬 기반 풀이를 자연스럽게 떠올리기

### 4주차
- 투 포인터
- 누적합
- 이분 탐색
- 목표: 시간 초과 나는 풀이를 개선하는 감각 만들기

### 5주차
- 우선순위 큐
- DFS / BFS
- 그리디
- 목표: Level 2 입문 문제 대응력 만들기

### 6주차
- 기본 DP
- 혼합 유형 복습
- 실전 모의 풀이
- 목표: 30~90분 제한 시간 안에 2~3문제 해결 훈련

## 문제 풀이 비율

- 이론 20%
- 손풀이/접근법 고민 20%
- 구현 40%
- 복기 20%

## 하루 학습 루틴

1. 이론 문서 1개 읽기
2. 관련 프로그래머스 문제 2~3개 풀기
3. 못 푼 문제는 30분 안에 막히면 힌트 확인
4. 풀이 후 반드시 "왜 이 풀이가 맞는지" 한 줄 요약
5. 실수 유형 기록

## 문제를 고를 때 기준

- 처음 2주: Level 1 위주
- 3~4주차: Level 1 빠르게 + 쉬운 Level 2 섞기
- 5주차 이후: Level 2 주력

문제를 고를 때는 난이도보다 유형을 먼저 맞추는 게 좋다.

예시:
- 문자열 공부 날: 문자열/구현 문제만 3개
- 스택 공부 날: 괄호, 올바른 순서, 제거/복원 계열 집중
- 이분 탐색 공부 날: 정답 범위를 기준으로 탐색하는 문제까지 확장

## 레포 활용 방식

### 이론
- `docs/` 문서를 먼저 읽는다.

### 코드
- `src/programmers/level1/`
- `src/programmers/level2/`
- `src/programmers/kit/`

위 구조를 기준으로 문제를 누적한다.

### 회고
- `프로그래머스 문제 풀이 템플릿.md` 또는 공통 템플릿을 사용한다.

## 실전에서 가장 중요한 판단

### 1. 문제를 읽고 무엇을 먼저 볼 것인가
- 입력 크기
- 정답으로 요구하는 것
- 중복 허용 여부
- 정렬 가능 여부
- 순서가 중요한지 여부

### 2. 풀이 방향을 정하는 기준
- 문자열/배열 구현인가
- 해시로 빈도나 존재 여부를 처리할 수 있는가
- 스택/큐로 순서를 관리해야 하는가
- 완전탐색이면 범위가 가능한가
- 시간 초과가 나면 투 포인터/누적합/이분 탐색으로 줄일 수 있는가

### 3. 합격권 기준에서 제일 치명적인 실수
- 문제를 덜 읽고 성급히 구현 시작
- 예외 케이스를 놓침
- 시간 복잡도 계산 없이 코드 작성
- 푼 문제를 복기하지 않음

## 내가 멘토링할 방식

- 문제를 풀기 전: 어떤 유형인지 같이 분류
- 풀이 중: 접근법이 맞는지 점검
- 제출 후: 코드, 시간 복잡도, 실수 포인트 피드백
- 반복 중: 약한 유형을 뽑아 다음 학습 순서 재설계

## 현재 권장 출발점

현재 레포 상태를 보면 문자열, 배열, 정렬, 투 포인터, 이분 탐색, 스택, 큐, 누적합 기초 자산이 이미 있다.

따라서 시작은 아래 순서가 가장 좋다.

1. 문자열
2. 배열
3. 해시
4. 스택
5. 큐
6. 정렬

이 구간을 먼저 끝내면 프로그래머스 Level 1과 쉬운 Level 2의 체감 난이도가 많이 내려간다.
